# 关键决策

- **关键决策**：在开发中使用看板工具，且是否使用 worktree 及使用哪个版本的 worktree 交由用户自由选择，同时要建立标准化流程。

  - **问题**：在自动化开发过程中，如何提高开发效率、协调多 agent 工作以及解决代码合并和项目管理的问题。
  - **讨论方案**：
  
    - 说话人 1 提出使用 agent board 工具监控 harness 能力，通过 Markdown 协调 agent 之间的工作，同时对 SPEC 的完善程度和开发过程中的指标进行量化监控。
    - 说话人 5 提出使用多个 worktree 同时开发多个 feature，将想法和功能点在 GitHub 上建 issue，分配到 workqueue 中开发，开发完成后提交 PR 并进行 code review。
    - 说话人 6 分享了自己在商业化项目中使用多 agent 开发的经验，包括规划 agent 和 coding agent 的分工，以及在开发过程中遇到的问题和解决方法。
  - **决策依据**：不同开发者有不同的习惯和需求，给予用户自由选择的权利可以更好地适应各种开发场景，同时建立标准化流程可以提高开发的效率和质量。

# 金句时刻

「很多人并发上不去，是因为还没有足够的去信任编码工具，要把 agent 当做团队里的真实成员，做好上下文构建，转变为管理者心态，才能释放注意力，提高并发量。」

—— 强调了信任编码工具和转变管理心态对提高开发并发量的重要性。



「一个人只有管理好一个 agent 才能管理好三个，管理好三个才能管理好 10 个，要循序渐进地把并发度拉上去，同时遵循 DRY 原则，避免重复劳动。」

—— 指出了提高并发度的方法和重要原则。



「在开发过程中，additions 和 deletions 占 additions 的 12% - 15% 是比较健康的，30% 以上可能发生局部或整体的 refactor，是不健康的，要通过量化指标监控项目。」

—— 提供了判断项目开发健康状况的量化指标。
